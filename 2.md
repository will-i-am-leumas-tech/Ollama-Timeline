# GOAL.functions.md — Functional Companion Spec
# Leumas Folder Timeline

## Purpose

This file exists to complement `GOAL.md` by defining the actual functions, modules, APIs, and frontend behaviors required to make the app real.

The first goal file explains the product.
This file explains the working logic.

The purpose is to ensure Codex does not just build a pretty UI, but actually builds the correct engine underneath it.

---

# Core Intent

The app should help us quickly answer questions like:

- What folders changed most recently?
- What projects have I been working on lately?
- What activity happened today, yesterday, or this week?
- What folders are most active?
- What project roots were touched recently?
- What work context should be sent to Ollama so it can answer questions about recent development activity?

This file defines the functions that make those features possible.

---

# Functional Outcomes We Need

## 1. Show what folders/projects changed most recently

This requires functions that can:
- scan folders
- detect modification timestamps
- aggregate folder-level activity
- sort folders by latest change
- return timeline-ready results

## 2. Show what we have been working on lately

This requires functions that can:
- infer likely project roots
- group file activity by folder/project
- summarize recent changes
- filter activity by date window

## 3. Build a summarized timeline of project activity

This requires functions that can:
- generate timeline events
- prevent duplicate spam events
- create readable summaries
- group events by day/time range

## 4. Build a searchable/indexed view of recent work

This requires functions that can:
- store indexed folders and file metadata
- search by folder name, path, extension, or recent event text
- paginate large result sets
- return lightweight results fast

## 5. Provide a future-ready context layer for Ollama

This requires functions that can:
- pull recent activity from the database
- compress it into useful context
- include only relevant summaries
- pass it into Ollama cleanly

14) IMPORTANT FUNCTIONAL RULES
Rule 1

Do not read full file contents during MVP scanning unless explicitly needed later.

Rule 2

Prefer folder-level summaries over giant raw file lists.

Rule 3

Ignore rules must be applied before recursion, not after.

Rule 4

Timeline summaries must be human-readable, not raw JSON dumps.

Rule 5

Ollama context must be compact and relevant, not huge.

Rule 6

Search results must be fast and paginated.

Rule 7

Watch mode should not spam the UI with one event per keystroke/save if files change rapidly.
